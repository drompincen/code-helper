3 Upload Flow

Client sends bytes → server stores in intake → artifact row created

Client
Read file bytes, detect media type
▶▶▶
gRPC: UploadFile
fileName, path, mediaType, bytes
▶▶▶
Server
Store in intake, compute SHA-256

What the Server Does

1
Generate artifact ID UUID
Unique identifier for this upload
2
Write to intake directory
temp/intake/<artifact-id>/<filename>
3
Compute SHA-256 checksum
For deduplication and integrity verification
4
Insert artifacts row STORED_IN_INTAKE
Metadata persisted to DuckDB

Upload-Dir (Batch)

How upload-dir works

Client scans the local directory recursively

Filters files by extension (e.g., .java,.md,.yml)

Uploads each file one at a time via UploadFile RPC

Prints summary: uploaded / skipped / failed

Key rule: No parsing during upload. Upload is fast and simple. Ingestion happens asynchronously later.
